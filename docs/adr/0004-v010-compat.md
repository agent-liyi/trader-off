# ADR-0004: v0.1.0 Backward Compatibility

## Status

Accepted

## Context

v0.2.0 must maintain backward compatibility with v0.1.0:
- Existing serialized models (`models/YYYYMMDD_HHMMSS/`) must load
- CLI commands `train|predict|backtest|feature-importance` must work unchanged
- `LGBMTop20Strategy` must remain importable and functional

We evaluated:

1. **Dual-format detection** in `load_model()`
2. **Migration script** to convert v0.1.0 models to v0.2.0 format
3. **Abandon v0.1.0 compatibility**

## Decision

We support **dual-format detection** in `load_model()`:

| Version format | Detection | Behavior |
|----------------|-----------|----------|
| `20260101_120000` (15 chars, v0.1.0) | `len==15 and version[8]=="_"` | Read v0.1.0 dir structure |
| `v0.2.0.5` | `^v\d+\.\d+\.\d+(\.incr\d+)?$` | Read v0.2.0 dir structure |
| Other | — | Raise `FileNotFoundError` |

v0.1.0 models loaded via `load_model()` have `metadata` fields filled with `None` for missing fields.

## Consequences

**Positive:**
- Zero migration cost for existing deployments
- v0.1.0 models continue to work without conversion
- Clear error messages for invalid version formats

**Negative:**
- Two directory structure styles coexist in `models/`
- Regular expression matching adds slight overhead to `load_model()`

**Mitigation:**
- v0.1.0 format checked only after v0.2.0 format fails
- NFR-1000 AC-1/AC-2 verify with real v0.1.0 fixture
- `OptimizedTopKStrategy` falls back to `LGBMTop20Strategy` when `weights.csv` missing/stale
